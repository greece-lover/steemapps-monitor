# Phase 4 — Öffentliches Dashboard (lokale Entwicklung)

**Datum:** 2026-04-24
**Phase:** 4
**Status:** abgeschlossen (inkl. Node-Listen-Erweiterung)

## Scope-Korrektur gegenüber ursprünglicher Phase-4-Planung

Ursprünglich war Phase 4 als „Deploy nach `api.steemapps.com`" formuliert.
Der Autor hat vor Beginn klargestellt: die VM bleibt **rein intern**, das
Dashboard wird lokal fertiggebaut und getestet, der öffentliche Deploy
landet später auf dem production-server. Diese Phase liefert deshalb:

- sqlite3-CLI auf der VM (Dev-Komfort)
- zwei neue API-Endpoints (`/history` mit Sparkline-Shape, neuer `/uptime`)
- vollständiges Dashboard-Frontend unter `frontend/` (Vanilla JS +
  Chart.js via CDN)
- Browser-Smoke-Test über SSH-Tunnel gegen die VM-API

Kein Reverse-Proxy, kein nginx, keine UFW-Änderung.

## Ergebnisse

### 1. sqlite3-CLI auf der VM

`sudo apt install sqlite3` → Version 3.45.1. Die Phase-3-Arbeitsanweisung
„ad-hoc-Queries bräuchten sonst Python-Wrapper" ist damit erledigt.

### 2. API-Erweiterungen (`api.py`)

- **`GET /api/v1/nodes/{url}/history?hours=1..168`** — jetzt mit
  sparkline-freundlicher Payload: Array von `{ts, latency_ms, success}`
  in chronologischer Reihenfolge (oldest first, damit Chart.js natürlich
  left-to-right zeichnet). Default `hours` von 24 auf 1 reduziert, weil
  das der Dashboard-Use-Case ist.
- **`GET /api/v1/nodes/{url}/uptime?days=1..30`** — neu. Liefert
  `{total, ok, uptime_pct}`. Wiederverwendet `database.get_uptime_stats`.
- **CORS-Middleware** (`allow_origins=["*"]`, `GET` only). Die Oberfläche
  ist read-only und ohne Auth, das Dashboard wird später von einer
  anderen Domain kommen.
- **DB-Path-Auflösung zur Laufzeit**: API ruft jetzt
  `database.get_*(…, db_path=database.DB_PATH)` explizit statt mit dem
  einmal-eingefrorenen Default-Arg auf — nötig, damit `test_api.py`
  über `monkeypatch.setattr(database, "DB_PATH", tmp)` isoliert werden
  kann.

### 3. API-Test-Suite (`tests/test_api.py`)

11 neue Tests mit FastAPI-`TestClient`, gegen eine Temp-SQLite-DB per
`pytest.tmp_path`:

- `/health` → liefert `methodology_version = mv1`
- `/status` mit leerer DB → alle Nodes `status=unknown, score=None`
- `/status` mit eingefügten Rows → Score-Regel greift (100 für sauber,
  30 für 3000 ms Latenz), `reference_block` = Max über erfolgreiche
  Heights, `block_lag` korrekt für laggy Node
- `/history` → Points oldest-first, lean-Shape (nur `ts/latency_ms/success`),
  `success` als Boolean (nicht 0/1)
- `/history` → 404 für unbekannten Node, 422 für `hours=0` und
  `hours>168`
- `/uptime` → prozent-korrekt bei gemischten OK/Fail-Messungen
- `/uptime` → 404 für unbekannten Node, 422 für `days=0` und `days>30`
- CORS-Header `Access-Control-Allow-Origin: *` ist gesetzt

Gesamtsuite lokal: **29/29 grün** (vorher 18). Auf der VM nach Deploy
ebenfalls 29/29.

### 4. Dashboard-Frontend (`frontend/`)

- `index.html` — semantisches Markup, Fonts via Google-CDN
  (Inter Tight + Fraunces), Chart.js via jsdelivr, kein Build-Step.
- `css/main.css` — Design-Sprache der Hauptseite: dark Background
  (`#0a0a0a`), Lime-Akzent (`#b7e34a`), Fraunces für Display-Zahlen
  (Score, Latenz, Uptime), Inter Tight für UI. Mobile-Breakpoint bei
  600 px — Node-Grid auf eine Spalte, Stat-Row von 4 auf 2 Spalten,
  Schriftgrößen leicht reduziert.
- `js/main.js` — Vanilla, kein Framework. `refresh()` ruft
  `/api/v1/status` alle 60 Sekunden, baut pro Node eine Card, hydriert
  danach pro Node `history?hours=1` (Sparkline) und `uptime?days=1` +
  `uptime?days=7`. Eine Chart.js-Instanz pro Canvas wird wiederverwendet
  (kein Listener-Leak). API-Basis-URL per Query-Param `?api=`
  überschreibbar, Default `http://localhost:8110`.
- `frontend/README.md` — Entwicklungs-Anleitung (SSH-Tunnel) und eine
  „Nicht produktionsreif"-Liste für den späteren production-Deploy
  (SRI-Pinning, CSP, Build-Time-API-URL).

### 5. SSH-Tunnel und Browser-Smoke-Test

- Hintergrund-Tunnel `ssh -L 8110:127.0.0.1:8110 -N steemfork` aufgebaut,
  `curl http://localhost:8110/api/v1/health` antwortet.
- Zweiter Hintergrund-Server: `python -m http.server 8777` unter
  `frontend/` — Chrome auf `file://` hätte CORS gegen das API-Root
  blockiert, der lokale Dev-Server löst das sauber.
- Chrome-Automation: Tab geöffnet auf `http://127.0.0.1:8777/index.html`,
  DOM nach einer Runde Rendering inspiziert:
  - 4 Node-Karten gerendert
  - Methodology-Version `mv1`, Reference-Block `105,473,995`
  - Scores: steemit 80, justyy 100, steem.fans 80, steemyy 100 (matcht
    exakt den aktuellen `/status`-Output)
  - Uptime-Zellen 24 h und 7 d beide 100.00 % über alle Nodes
  - Sparkline-Canvas in jeder Karte vorhanden
  - Kein Error-Banner, keine Console-Errors
- Mobile-Breakpoint: Chromes `resize_window` klammert auf >1000 px auf
  dieser Maschine — stattdessen die 8 Mobile-CSS-Rules direkt per
  `!important`-Probe-Stylesheet geladen und die Layout-Änderung
  verifiziert: Grid wird einspaltig, Stat-Row wird 2×2, keine
  horizontale Overflow. Rules sind syntaktisch und semantisch korrekt.

## Bewusste Design-Entscheidungen

- **`/history` schema-Bruch (rows → points) statt additives Feld.** Die
  API ist noch nicht öffentlich, und Sparkline-Daten sind der einzige
  aktuelle Konsument. Zwei Felder parallel zu pflegen wäre teurer als
  ein sauberer Rename jetzt. Dokumentiert in `docs/API.md`.
- **CORS auf `*` statt auf eine Allow-List.** Die API ist read-only, es
  gibt keine Cookies, keine Auth. Eine Allow-List wäre Security-Theater.
  Wenn die Chain-Post-Feature (Phase 5) State-Changing-Endpoints dazu-
  bringt, wird das neu bewertet.
- **Chart.js via CDN, nicht gebündelt.** Für die Dev-Phase völlig ok,
  spart einen Build-Step. Die `frontend/README.md` listet das als
  „vor production-Deploy zu fixen" (SRI-Pinning oder self-hosted).
- **Zwei separate Uptime-Calls (`days=1` und `days=7`) statt einem
  batch-Endpoint.** Weniger Code auf beiden Seiten, und der Server
  cached gar nichts — die zweite Abfrage ist <1 ms auf SQLite.
- **Dashboard-Entwicklung über SSH-Tunnel, keine UFW-Änderung.** Matcht
  Phase-3-Policy „VM intern". Sobald der production-Deploy ansteht, wird das
  neu bewertet und nginx kommt an die Stelle des Tunnels.
- **DB-Path-Expliziterung statt Default-Arg-Reset.** Eleganter wäre
  gewesen, `get_recent_measurements(db_path=None)` zu schreiben und
  innen `DB_PATH` zu resolven. Für Phase 4 reicht der explizite Argumen-
  tenpfad aus — die API ist der einzige Konsument, der das betrifft.

## Nachtrag: Node-Liste von 4 auf 10 erweitert

Der Autor hat acht weitere Nodes in `nodes.json` ergänzt. Zwei davon
mussten raus:

- **`api.dlike.io`** — HTTP 308 in eine Self-Redirect-Schleife
  (`curl: Maximum (50) redirects followed`). Nicht nutzbar als
  JSON-RPC-Endpoint.
- **`api.boylikegirl.club`** — HTTP 302 → `/swagger`, dort ein
  OpenAPI-Dokumentations-Endpoint (HTTP 404 bei GET, 411 bei POST).
  Keine condenser_api-Implementierung, sondern eine REST-API-Doku-Seite.

Die übrigen sechs wurden einzeln gegen `condenser_api.get_dynamic_global_properties`
getestet — alle antworten HTTP 200 mit validem `head_block_number` und
sind innerhalb eines Blocks synchron:

| Node | Region | Latenz (erster Probe) |
|---|---|---|
| api.moecki.online | eu-central | 85 ms |
| steemd.steemworld.org | eu-central | 184 ms |
| api.campingclub.me | asia | 260 ms |
| steem.senior.workers.dev | global | 422 ms |
| api2.steemyy.com | global | 572 ms |
| steem.justyy.com | us-central | 720 ms |

Die aktive Liste enthält damit 10 Nodes. Aktualisierte `nodes.json`
per `scp` auf die VM übertragen, Service neu gestartet, erster
Tick-nach-Restart `10/10 ok`, alle Nodes auf Block 105474709. Die
Liste im Repo dokumentiert den aktuellen Messumfang.

Gefunden beim Prüfen: **`steemd.steemworld.org` ist der schnellste
Node (180 ms) und deutlich flotter als `api.steemit.com` (751 ms)
vom gleichen Messpunkt aus.** Das ist die Art von Aussage, die der
öffentliche Report später tragen wird — gut, dass die Methodik
dafür jetzt steht.

## Offene Punkte → Phase 5 (Daily-Report-Generator auf die Chain)

Phase-5-Fokus laut Konzept: einmal pro Tag eine Steemit-Post + eine
`custom_json`-Operation `steemapps_api_stats_daily` auf die Chain
schreiben. Dafür zu klären:

1. **Reporter-Account-Name** — `@steem-api-health` vs. `@greece-lover`.
   Aus Konzept weiterhin offen.
2. **Signieren via** (a) SARH-crypto-Library (schon implementiert),
   (b) `beem` als externe Dependency, (c) `steem-python`? SARH wäre die
   sauberste Wiederverwendung.
3. **Key-Handling** — Posting-Key im `.env.local` mit `chmod 600`;
   das Service-Unit-Template `EnvironmentFile=` muss ergänzt werden.
4. **Cron-Zeitfenster** — Konzept sagt 02:00 MESZ, das sind 00:00 oder
   01:00 UTC je nach Sommerzeit. Besser einmal pro UTC-Tag um 00:30 UTC.
5. **Post-Template** — HTML-Body mit Uptime-Tabelle, Vergleich zur
   Vorwoche, Link zu `frontend/index.html` (später
   `dashboard.steemapps.com`).
6. **Chain-`custom_json`-Shape** — verbindliches Schema, damit externe
   Konsumenten (Welako, Third-Party-Dashboards) darauf bauen können.
