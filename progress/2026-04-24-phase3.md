# Phase 3 — Monitor-Kern-Implementierung

**Datum:** 2026-04-24
**Phase:** 3
**Status:** Code fertig, Deploy pausiert bis VM wieder erreichbar ist

## Ziel

Ein funktionsfähiger, lokaler Monitor auf `/opt/steemapps-monitor/` der VM,
der alle 60 Sekunden die vier konfigurierten Steem-API-Nodes abfragt,
Messwerte in SQLite persistiert und einen lokalen HTTP-Endpoint auf
`127.0.0.1:8110` bereitstellt.

## Ergebnisse (lokal, committet)

### Neue Module im Repo-Root

Flaches Layout, keine Package-Struktur — die Modul-Menge ist klein und
stabil genug, dass ein zusätzliches Verzeichnis nur Zeremonie wäre.

- `monitor.py` — Entry-Point. Startet asyncio-Event-Loop mit zwei Tasks:
  der Poll-Schleife und einem eingebetteten uvicorn-Server. Die
  Poll-Schleife feuert sofort bei Boot einen ersten Tick (damit die API
  nicht erst nach einer Minute Daten hat), dann alle 60 s. Signal-Handler
  für SIGINT/SIGTERM, damit systemd sauber stoppen kann.
- `database.py` — SQLite-Schicht. `measurements`- und `nodes`-Tabelle mit
  Indexen auf `timestamp`, `node_url` und dem zusammengesetzten
  `(node_url, timestamp)`. WAL-Modus für den Ein-Schreiber-Viele-Leser-
  Pattern. Funktionen: `initialise`, `sync_nodes`, `insert_measurement`,
  `get_recent_measurements`, `get_latest_per_node`, `get_uptime_stats`,
  `row_count`.
- `scoring.py` — reine Score-Berechnung, keine I/O. Setzt die
  `mv1`-Regeln aus `MEASUREMENT-METHODOLOGY.md` 1:1 um: Latenz-Bänder
  (>500 ms, >2000 ms), Block-Lag-Bänder (>3, >10), Error-Rate >20 %
  über letzte 20 Ticks, kein Response in diesem Tick. Penalty-Konstanten
  sind modul-level und matchen das Konzept. Scores werden bei 0 gekappt.
- `api.py` — FastAPI-App. Drei Routen unter `/api/v1/`: `health`,
  `status`, `nodes/{url}/history`. `status` verknüpft den letzten Tick pro
  Node mit dem Score und einer menschenlesbaren `reasons`-Liste
  (genau die Penalty-Labels, die der Score angewendet hat).
- `logger.py` — stdout-Logging mit ISO-Zeitstempel für systemd-Journal.
  Idempotentes `setup()`.
- `config.py` — Pfade, Intervalle, `load_nodes()`-Helper. `SOURCE_LOCATION`
  kommt aus `STEEMAPPS_SOURCE_LOCATION` env var (Default
  `contabo-de-1`), damit später Multi-Location-Betrieb ohne Code-Änderung
  geht.
- `nodes.json` — die vier bestätigten Nodes aus Phase 1:
  `api.steemit.com`, `api.justyy.com`, `api.steem.fans`, `api.steemyy.com`.
  Einzige Wahrheitsquelle; `sync_nodes` übernimmt Upsert + Markierung
  entfernter Nodes als inaktiv.

### systemd-Unit

`deploy/steemapps-monitor.service` mit User `holger`, WorkingDirectory
`/opt/steemapps-monitor`, `Restart=always`, `After=network-online.target`
und einem robusten Hardening-Block:

- `NoNewPrivileges`, `PrivateTmp`, `ProtectSystem=strict`, `ProtectHome`,
  `ReadWritePaths=` nur `data/` und `logs/`
- `RestrictAddressFamilies=AF_INET AF_INET6 AF_UNIX`, keine raw sockets
- `MemoryDenyWriteExecute`, `LockPersonality`

`StandardOutput=journal` macht `journalctl -u steemapps-monitor` zum
One-Stop-Logbuch.

## Bewusste Design-Entscheidungen

- **Poller und API in einem Prozess, nicht zwei.** Für Phase 3 ist die
  zusätzliche Inter-Prozess-Koordination (DB-Mutex, Cross-Process-SIGHUP)
  nicht gerechtfertigt. Wenn nginx in Phase 4 dazukommt, wird die API in
  eine eigene Unit ausgegliedert — dann ist es aber auch lohnend.
- **FastAPI-Routen unter `/api/v1/`.** Das Konzept-Dokument schlägt die
  Pfade ohne Prefix vor (`/status`, `/nodes/…`); der Prefix macht aber
  das spätere Versionieren von Schema-Änderungen ohne Breaking-Change
  einfacher — und nginx-Rewrite-Regeln für Phase 4 werden trivialer,
  wenn das API-Root klar abgegrenzt ist.
- **Block-Lag-Referenz = `max(head_block)` über **erfolgreiche** letzte
  Ticks aller Nodes.** Wenn ein Node fehlschlägt, zählt seine NULL nicht
  als „Referenz 0". Das matcht die Methodik-Doku und verhindert, dass ein
  einzelner Ausfall die Lag-Bewertung aller anderen kippt.
- **`source_location` in jeder Zeile, default `contabo-de-1`.** Kostet
  heute nur eine Spalte und macht Phase-5-Multi-Location-Auswertung
  trivial: einfach Messungen aus verschiedenen Instanzen in dieselbe DB
  schreiben (oder per-Instance-DBs mergen, je nach Entscheidung).
- **`sync_nodes` markiert entfernte Nodes als `active=0`, löscht nicht.**
  Historische Messungen bleiben joinable. Wenn ein Node später wieder
  reinkommt, flippt das Flag zurück — Null Datenverlust.
- **`/api/v1/status` liefert `reasons` direkt mit.** Das ist keine interne
  Debug-Info, sondern Teil des Methodik-Versprechens „jede Zahl ist
  reproduzierbar": der User sieht genau, welche Regel welche Punkte
  abgezogen hat.
- **Keine externe Anbindung in Phase 3.** Port 8110 nur auf 127.0.0.1,
  keine UFW-Änderung, keine nginx-Regel. Das passt zur Tabu-Liste
  (keine Fremdeingriffe in bestehende Dienste) und vermeidet Angriffs-
  fläche während der Beta.

## Offene Punkte → Phase 3 Wrap-up (nach VM-Rückkehr)

1. Repo auf `/opt/steemapps-monitor/` klonen (ist bereits leer angelegt)
2. venv erstellen, `pip install -r requirements.txt`
3. `sudo cp deploy/steemapps-monitor.service /etc/systemd/system/`,
   `daemon-reload`, `enable --now`
4. 30–60 Minuten laufen lassen
5. DB-Stand prüfen: `sqlite3 data/measurements.sqlite "SELECT COUNT(*) FROM measurements"`
6. API abfragen: `curl -s 127.0.0.1:8110/api/v1/status | jq`
7. Abschluss-Commit mit den tatsächlichen Messwerten aus der Laufzeit
